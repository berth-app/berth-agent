Berth Agent v0.4.3

Lightweight deployment agent for Linux servers. Receives code from the Berth macOS app, runs it, and reports back.

Contents: Overview Installation Connection Modes Pairing Directories Configuration Commands Security Self-Upgrade Systemd Troubleshooting

Overview

The Berth Agent (berth-agent) is a single Rust binary that runs on Linux servers as a systemd service. It has zero runtime dependencies — everything is statically linked.

The agent:

  • Receives code deployments from the Berth desktop app or CLI
  • Runs projects with automatic dependency installation
  • Streams logs back in real-time
  • Maintains execution history in a local SQLite database
  • Executes scheduled jobs independently (even when the desktop app is offline)
  • Publishes services to public URLs via cloudflared tunnels
  • Self-upgrades with automatic rollback on failure

Installation

Quick Install

curl -sSL https://agent.getberth.dev/install.sh | sudo bash

The installer will guide you through setup:

1

System setup

Detects your architecture (x86_64 or aarch64), creates a dedicated berth system user, and downloads the binary.

2

Connection mode

Choose how the agent communicates with the desktop app:

  • Synadia Cloud (NATS) — Recommended. Paste your NATS credentials. Both sides connect outbound — zero inbound ports, works behind NAT.
  • Direct connection (mTLS) — Generates TLS certificates. Requires port 50051 to be reachable.
3

Service start

Installs a rollback script, creates the systemd service, and starts the agent.

Verify

sudo systemctl status berth-agent
sudo journalctl -u berth-agent -f

Uninstall

curl -sSL https://agent.getberth.dev/install.sh | sudo bash -s -- --uninstall

Connection Modes

Synadia Cloud (NATS) — Recommended

Both the desktop and agent connect outbound to Synadia's global NATS servers. Neither needs to accept incoming connections.

Your Mac Synadia Cloud Linux Server | | | |--- connect outbound (TLS) ------->|<------ connect outbound (TLS) ------| | | | |--- send command ----------------->|----------- forward ---------------->| |<-- stream response ---------------|<---------- stream back -------------|

Setup

  1. Create an account at cloud.synadia.com
  2. Create a System → Account → User (or use defaults)
  3. Go to Users → select user → "Copy Credentials"
  4. Paste when the installer prompts, or configure later: 
    sudo -u berth berth-agent setup-nats

Credentials are saved to ~/.berth/nats.creds (0600 permissions).

Direct Connection (gRPC + mTLS)

The desktop connects directly to the agent's IP address. Port 50051 must be reachable.

  1. During install, choose "Direct connection" — certificates are auto-generated
  2. Copy ~/.berth/certs/ca.crt, client.crt, client.key to your Mac
  3. Import them in the Berth app when adding the target

Pairing

Before accepting commands, the agent must be paired with your Berth desktop app. Pairing establishes a cryptographic shared secret used to sign all future commands.

1

Get the pairing code

When the agent starts unpaired, it generates an 8-character code:

sudo journalctl -u berth-agent -f | grep PAIRING

Output: PAIRING CODE: ABCD1234 (expires in 5 minutes)

2

Enter in the Berth app

Go to Targets → Add Target → enter the pairing code when prompted.

3

Verification & secret exchange

The app and agent perform a challenge-response handshake. On success, a 256-bit shared secret is generated and stored on both sides (agent SQLite + Mac Keychain).

Pairing Details

PropertyValue
Code length8 characters (A-Z, 2-9, no I/O/L/0/1)
Expiry5 minutes
Rate limiting5 failed attempts → 5-minute cooldown
SecurityChallenge-response (HMAC of agent challenge using code as key)

Re-pairing

To pair with a different desktop app, restart the agent for a new code:

sudo systemctl restart berth-agent
sudo journalctl -u berth-agent -f | grep PAIRING

Security

Command Signing

Every command sent to the agent is signed with HMAC-SHA256 using the shared secret established during pairing:

  • Each command includes a signature, nonce, and timestamp
  • Commands older than 60 seconds are rejected
  • Nonces prevent replay attacks (10K nonce tracker)
  • The agent rejects all commands before pairing is complete

Path Safety

  • Path traversal (../), absolute paths, and null bytes are rejected
  • Tar archive entries are validated during extraction
  • Deploy directories are sandboxed to ~/.berth/deploys/

Process Isolation

  • Runs as dedicated berth user (not root)
  • systemd NoNewPrivileges=true and ProtectSystem=strict
  • Credentials and keys have 0600 permissions (owner-only)
  • Environment variable values are passed at runtime, never persisted on the agent

Best practices:

  • Use NATS mode to avoid exposing ports
  • Keep the agent updated (berth-agent update)
  • Monitor logs for unauthorized pairing attempts
  • If using direct mode, use firewall rules to restrict port 50051

Self-Upgrade

# Upgrade to latest
sudo -u berth berth-agent update

# Upgrade to specific version
sudo -u berth berth-agent update --version 0.4.3 --yes

How It Works

  1. Downloads new binary from GitHub releases
  2. Verifies SHA256 checksum
  3. Backs up current binary to berth-agent.old
  4. Atomic swap: new binary replaces current
  5. Exits with code 42 → systemd restarts with new binary
  6. Probation (30s): New binary must pass 3 health checks
  7. Failure: rollback script automatically restores backup

Check current version:

berth-agent --show-version

Troubleshooting

Agent won't start

sudo journalctl -u berth-agent -n 50
ls -la /home/berth/.berth/bin/berth-agent
ls -la /home/berth/.berth/agent.env

Cannot pair

  • Check logs for PAIRING CODE — it must be an unpaired agent
  • Codes expire in 5 minutes — restart for a new one
  • 5 failed attempts trigger a 5-minute cooldown
  • Ensure NATS credentials are valid on both sides

NATS not connecting

  • Verify nats.creds exists and has 0600 permissions
  • Check URL in agent.env: should be tls://connect.ngs.global
  • Test connectivity: curl -s https://connect.ngs.global
  • Reconfigure: sudo -u berth berth-agent setup-nats

Deployments failing

  • Check disk space: df -h /home/berth
  • Verify runtime: python3 --version, node --version
  • Check deploys dir: ls -la /home/berth/.berth/deploys/

Cannot publish (public URL)

  • Install cloudflared: apt install cloudflared or download from Cloudflare
  • Ensure the project is running and listening on the specified port

Upgrade failed

  • Rollback should be automatic via systemd ExecStopPost
  • Manual rollback: cp ~/.berth/bin/berth-agent.old ~/.berth/bin/berth-agent && sudo systemctl restart berth-agent
  • Check: ls -la ~/.berth/.upgrading ~/.berth/.rollback-count

Process stuck

  • From desktop: berth stop my-project --target my-server
  • Restart agent: sudo systemctl restart berth-agent (kills child processes)
